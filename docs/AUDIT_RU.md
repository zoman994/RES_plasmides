# Архитектурный аудит PlasmidVCS: 10 критических находок

**Дата:** 28 марта 2026
**Кодовая база:** 15 500+ строк, 28 React-компонентов, 22 Python-модуля

---

## Резюме

PlasmidVCS — работающая система с адекватным разделением React/Python. Но **три архитектурных решения активно тормозят масштабирование**: весь state в одном файле App.jsx на 646 строк, калькулятор Tm по формуле Wallace с погрешностью ±5°C, и ноль тестов на фронтенде. Исправление этих трёх вещей превращает прототип в продакшн-инструмент. Остальные семь находок — от важных оптимизаций до инвестиций в будущее, реализуемых за 8–12 недель.

## 1. App.jsx на 646 строк надо разобрать — миграция на Zustand

### Что сейчас

Весь state приложения — `fragments[]`, `junctions[]`, `primers[]`, `assemblies[]`, флаги UI — живёт в `useState` внутри App.jsx. 28 дочерних компонентов. Любое изменение state → полный re-render дерева. React ловит неизменённые DOM-ноды при reconciliation, но все 28 функций компонентов перевыполняются на каждое нажатие клавиши — пересоздаются замыкания, объекты, все inline-вычисления.

### Почему это проблема

Архитектура ломается на трёх порогах, которые PlasmidVCS уже перешагнул: больше 5–8 доменов state в одном компоненте, больше 150–200 строк логики state (у нас 4×), дерево компонентов глубже 3 уровней с общим state. TkDodo (мейнтейнер Zustand) писал что фиксил «огромную проблему производительности, перенеся useState + Context на Zustand» при том же объёме кода.

### Решение: Zustand v5 + slices + Immer

Zustand (~1 КБ gzipped) даёт подписки на основе селекторов: компонент ре-рендерится только когда меняется его кусок state. Структура — пять слайсов: `fragmentSlice`, `junctionSlice`, `primerSlice`, `assemblySlice`, `uiSlice`, каждый в своём файле. Immer middleware — мутабельный стиль обновлений на вложенных ДНК-структурах. `devtools` — Redux DevTools дебаг. `persist` с `partialize` — автосохранение доменных данных (не UI) в localStorage/IndexedDB. `subscribeWithSelector` — внешние реакции: автосохранение, триггеры пересчёта праймеров.

Почему не Context? Context ре-рендерит ВСЕХ подписчиков при изменении любого значения. Пять Context-ов — boilerplate без автоматической оптимизации Zustand. Почему не Jotai? Атомарная модель Jotai хороша для сложных графов зависимостей; доменная slice-архитектура PlasmidVCS лучше ложится на top-down store Zustand.

### Нормализация state

Перевести `fragments[]` массивы в `{ byId: Record<string, Fragment>, allIds: string[] }`. Это даёт O(1) поиск для ссылок junction→fragment (сейчас O(n) через `.find()`), убирает дублирование данных, гарантирует единый источник правды при обновлении имени фрагмента, на который ссылаются несколько стыков.

### React Compiler v1.0

Установить немедленно через `babel-plugin-react-compiler@latest`. Автоматически мемоизирует компоненты и значения на этапе сборки — убирает ручные `useMemo`/`useCallback`/`React.memo`. Meta задеплоил его на Instagram. Но компилятор оптимизирует ре-рендеры внутри существующей архитектуры — он не чинит монолит на 646 строк. Поставить компилятор за 1–2 часа, потом мигрировать на Zustand за 2–3 дня.

**Приоритет: критический.** Самое влиятельное изменение.

## 2. Координаты праймеров: нужен prefix sum массив вместо пересчёта на лету

### Что сейчас

Праймеры хранят позиции относительно родительского фрагмента. Перевод в абсолютные позиции для Sequence View разбросан по разным местам — одни компоненты пересчитывают, другие используют устаревшие значения. Часть расчётов через `useEffect` пишущий в отдельный state — задокументированный React-антипаттерн, двойной рендер и задержка.

### Решение: мемоизированный Zustand-селектор с prefix sum

Prefix sum — та самая структура данных, которую геномные браузеры (IGV.js, Gosling.js) используют для координатных трансформов:

```javascript
const offsets = [0];
for (let i = 0; i < fragments.length; i++) {
  offsets[i + 1] = offsets[i] + fragments[i].length;
}
```

Для 57 фрагментов — тривиально быстро, но паттерн масштабируется до тысяч. Экспонировать как мемоизированный селектор из Zustand store — пересчитывается только при изменении `fragments`, общий для всех потребителей.

### Для больших последовательностей (34 МБ+): с localStorage на IndexedDB

localStorage — лимит 5–10 МБ, блокирует main thread синхронно, хранит только строки. IndexedDB — гигабайты асинхронно, нативный `ArrayBuffer`. Обёртка: **Dexie.js** (100K+ сайтов, React hooks через `dexie-react-hooks`). Сжатие через нативный `CompressionStream` API — ДНК-последовательности достигают **70–75% gzip-сжатия** (алфавит из 4 символов), или 75% через 2-битную кодировку (4 основания на байт в `Uint8Array`).

**Приоритет: высокий.** Один prefix sum селектор убирает целый класс багов консистентности.

## 3. При 113 компонентах мемоизация важнее виртуализации

### Что сейчас

57 PartBlock + 56 JunctionBlock — все ре-рендерятся при любом изменении state. Пользователь видит подтормаживание при редактировании.

### Диагноз

Проблема — ре-рендеры, не количество DOM-элементов. При 113 элементах и SVG, и DOM работают нормально — SVG деградирует заметно только выше 500 элементов, Canvas становится нужен около 1000+.

### Решение — три шага по порядку

**Шаг 1: Гранулярные Zustand-селекторы** (бесплатно, максимальный эффект). Каждый PartBlock подписывается только на данные своего фрагмента: `const fragment = useStore(state => state.fragments.byId[id])`. Редактирование праймера больше не ре-рендерит блоки фрагментов.

**Шаг 2: React.memo на PartBlock и JunctionBlock** (бесплатно). В сочетании с React Compiler становится автоматическим.

**Шаг 3: @tanstack/react-virtual с `horizontal: true`** (только если профилирование показывает проблемы). TanStack Virtual — headless hook с полным контролем DOM, поддержкой переменной ширины, динамическим измерением. React Flow использует аналогичный подход — viewport culling через `onlyRenderVisibleElements`, поверх внутреннего Zustand store.

**Для drag-and-drop: dnd-kit** (~4.5 КБ gzipped) с паттерном `useRef` + `requestAnimationFrame`. Во время перетаскивания — обновление позиции через CSS `transform` на ref (минуя React render цикл); коммит в Zustand state только при drop.

**Приоритет: высокий** (шаги 1–2), **средний** (шаг 3, только по результатам профилирования).

## 4. Калькулятор Tm ошибается на ±5°C — заменить на primer3-py

### Что сейчас

`calc_tm` использует простую формулу — скорее всего правило Wallace (2°C на AT + 4°C на GC) или базовую salt-adjusted формулу. Wallace никогда не предназначался для праймеров длиннее 13 нуклеотидов, игнорирует стэкинг-взаимодействия (nearest-neighbor эффекты), предполагает 1М NaCl. Для типичного 20-мера ошибка **±5°C** по сравнению с методом SantaLucia 1998.

### Почему это критично

5°C ошибки Tm → ПЦР-фейлы и фейлы сборки. Парные праймеры не отжигаются при одной температуре, появляются неспецифические продукты, стыки Golden Gate или Gibson не работают.

### Решение: primer3-py v2.3.0

Вышел январь 2026, MIT-лицензия, Python 3.8–3.14. Обёртка C-библиотеки Primer3 через Cython, термодинамическая модель SantaLucia 1998 NN — золотой стандарт, используемый NEB Tm Calculator, IDT OligoAnalyzer, NCBI Primer-BLAST. ~1000× быстрее subprocess-обёрток (~4.74 мкс на вызов). Также даёт анализ шпилек, гомодимеров и гетеродимеров:

```python
from primer3 import ThermoAnalysis
ta = ThermoAnalysis(mv_conc=50, dv_conc=1.5, dntp_conc=0.6, dna_conc=50.0)
tm = ta.calc_tm("GTAAAACGACGGCCAGT")  # → 49.16°C (±1°C от экспериментальных)
```

### Обработка кольцевых конструктов

Моделировать стык N-к-0 как полноценный junction-объект. Модулярная арифметика (`pos % len(sequence)`) для всех координатных расчётов. Типичные off-by-one ошибки: забыть закольцовывающий overlap (N-1 вместо N), неправильный спейсер между сайтом узнавания BsaI и позицией разреза (1 п.н. для BsaI, 2 п.н. для BbsI), inclusive vs exclusive индексация границ overlap.

**Приоритет: критический.** Неточный расчёт Tm подрывает ключевую ценность инструмента.

## 5. Golden Gate: овехенги нужно подбирать по данным Potapov

### Что сейчас

GG-реализация не использует опубликованные данные о fidelity лигирования. Без них подбор овехенгов опирается на устаревшие эвристики, которые по оценке самого NEB «излишне рестриктивны».

### Ключевой датасет

Potapov et al. (2018), «Comprehensive Profiling of Four Base Overhang Ligation Fidelity by T4 DNA Ligase», ACS Synth. Biol. 7(11):2665-2674. PacBio SMRT-секвенирование, частота лигирования для **всех 256 возможных 4-нуклеотидных овехенгов**, нормализованных к 100 000 событий на эксперимент. Датасет позволяет предсказать fidelity сборки для любого набора овехенгов как произведение индивидуальных fidelity стыков.

### Интегрировать MIT-библиотеки Edinburgh Genome Foundry

- **`tatapov`**: программный доступ к данным Potapov как pandas DataFrames. Поиск попарных частот лигирования для любой пары овехенгов.
- **`GoldenHinges`**: подбор овехенгов через constraint solver — без палиндромов, без комплементов, расстояние Хэмминга ≥2, GC 25–75%, запрещённые пары из tatapov.
- **`DNA Cauldron`**: симуляция сборки in silico — проверка дизайна до генерации протоколов.

### Опубликованная успешность по числу фрагментов

- **≤10**: ~100% fidelity, минимальный скрининг
- **12**: 99% правильных сборок
- **20**: ~98–100% предсказанная fidelity
- **24**: >90%, ~10× меньше колоний чем 12 фрагментов
- **35**: ~71%
- **52**: 49% (показанный максимум, нужна фаговая селекция)

Реализовать предупреждения: осторожность выше 12 фрагментов, строгое предупреждение выше 24, показывать предсказанный fidelity score из tatapov.

**Приоритет: высокий.** Fidelity овехенгов — разница между 99% и 40% успешности.

## 6. Тяжёлые вычисления — в Web Workers

### Что сейчас

Последовательности хранятся как plain strings в React state и localStorage. Анализ больших последовательностей блокирует main thread.

### Три независимых изменения

**Web Workers через Comlink.** Google Comlink (~1.3M еженедельных npm-загрузок) — воркеры ощущаются как async-вызовы функций. С `vite-plugin-comlink` — zero-config для Vite. Создать singleton `SequenceAnalysisWorker` с методами `calculateTm()`, `findORFs()`, `scanRestrictionSites()`, `translateSequence()`. Main thread остаётся отзывчивым. Для последовательностей до 100 КБ (типичные плазмиды) стандартный `postMessage` со structured cloning достаточен; `SharedArrayBuffer` — для геномного масштаба.

**Сжатие для IndexedDB** через нативный `CompressionStream` API. ДНК-текст — ~70–75% gzip-сжатие. 2-битная кодировка (A=00, C=01, G=10, T=11) — 4 основания на байт в `Uint8Array`: плазмида 10 000 п.н. = 10 КБ → 2.5 КБ.

**Готовые библиотеки.** `@teselagen/sequence-utils` (от команды Open Vector Editor) — комплемент, трансляция, рестрикционный анализ. Interval tree (`@flatten-js/interval-tree`) — O(log n + k) запросы «какие фичи пересекают позицию X».

**Приоритет: средне-высокий.**

## 7. Протоколы: стадийная DAG-модель вместо плоского списка

### Что сейчас

Протоколы — плоские списки шагов. Реальные клонирования имеют параллельные операции (несколько ПЦР), условные ветвления (результаты colony screening), многодневные зависимости.

### Стандарта нет

SBOL3 описывает дизайны, не протоколы. Autoprotocol (Transcriptic/Strateos) — самый зрелый машиночитаемый формат, но разработка похоже заглохла. LabOP (бывший PAML) — самый амбициозный стандарт (UML Activity Diagrams + RDF/OWL), но ранняя стадия. OpenTrons — плоские Python-скрипты без нативной многостадийности. Hamilton VENUS — проприетарные `.med` файлы.

### Рекомендация: Staged DAG в JSON

Именованные стадии верхнего уровня (человекочитаемые, линейная концептуальная прогрессия) содержат шаги, образующие DAG (параллельные операции внутри стадии). Между стадиями: зависимости, временные ограничения (стиль Autoprotocol: `"60:minute"`, `"37:celsius"`), ручные контрольные точки.

Цели экспорта по приоритету: (1) человекочитаемый Markdown/PDF — всегда доступен, робот не нужен; (2) Opentrons Python API v2 — крупнейшая доступная база пользователей; (3) PyLabRobot — hardware-agnostic; (4) Autoprotocol JSON для облачных лабораторий.

**Приоритет: средний.**

## 8. CRISPR: CRISPRscan вместо ML-моделей

### Контекст

PlasmidVCS — инструмент клонирования, не специализированная CRISPR-платформа. Нужен базовый дизайн gRNA, не полногеномный off-target анализ.

### Ключевой выбор

Doench Rule Set 2 (Azimuth) требует предобученных весов scikit-learn — нельзя реализовать только по статье. Gradient-boosted regression tree на ~4000 sgRNA. Оригинальный код — Python 2. В `crisprScore` (Bioconductor) отмечено что Azimuth «больше не поддерживается».

### CRISPRscan — оптимальный выбор

Линейная регрессия с ~30 позиционно-специфичными коэффициентами, опубликованными в supplementary materials (Moreno-Mateos et al., 2015). Реализация: **~50 строк чистого Python без зависимостей** — словарь + взвешенная сумма.

### Двухуровневая архитектура

**Уровень 1 (выпустить первым):** Эвристический фильтр — поиск PAM (все NGG сайты), GC 30–70%, отклонить TTTT (терминатор Pol III), предпочесть G на позиции 20, штрафовать C на позиции 3. Ноль зависимостей.

**Уровень 2 (выпустить вторым):** CRISPRscan линейный скоринг по опубликованным коэффициентам. Рецензированная оценка эффективности в ~50 строках.

**Для полногеномного off-target анализа** — перенаправлять пользователей на CRISPOR или CHOPCHOP. Не наша задача.

**Приоритет: низко-средний.**

## 9. Форматы файлов: импорт SnapGene и интеграция Benchling API

### SnapGene .dna

Проприетарный бинарный формат реверс-инженерен. Две Python-библиотеки: `snapgene_reader` (Edinburgh Genome Foundry, MIT, читает в BioPython SeqRecord) и встроенный `Bio.SeqIO.SnapGeneIO` (с BioPython v1.75). Файлы SnapGene хранят то, чего нет в GenBank: праймеры с цветами, настройки отображения фичей, двунаправленные фичи, история клонирования.

### Benchling API

Полноценный публичный REST API v2 с официальным Python SDK (`benchling-sdk`). Создание, чтение, обновление ДНК-последовательностей с нуклеотидами, аннотациями, праймерами, флагом `is_circular`. Аутентификация через API-ключи. Двусторонняя синхронизация: экспорт конструктов в Benchling registry, импорт последовательностей из Benchling.

### APE-файлы

Это GenBank с расширениями — `/ApEinfo_fwdcolor`, `/ApEinfo_revcolor` хранят цвета отображения. BioPython парсит как GenBank; извлечь `/ApEinfo_*` qualifiers для сохранения цветов.

### SBOL3 vs GenBank

SBOL3 захватывает design intent, иерархическую композицию, комбинаторный дизайн, провенанс — то, что GenBank не может. `pySBOL3` (v1.2+) зрелая, `sbol-utilities` — офлайн конвертация GenBank↔SBOL3. Но SBOL3 пока нишевый — поддерживать как перспективный формат экспорта, не основной.

### Многозаписные GenBank

Всегда `SeqIO.parse()` (итератор) вместо `SeqIO.read()`. При нескольких записях — диалог выбора с именами, длинами, топологией.

### Приоритет реализации форматов

- **Tier 1** (обязательно): GenBank (готово), FASTA, SnapGene .dna импорт, APE импорт, автодетекция формата
- **Tier 2** (желательно): Benchling API, EMBL, многозаписный GenBank
- **Tier 3** (будущее): SBOL3 экспорт, GFF3, пакетные операции

**Приоритет: средний.** Один только импорт SnapGene открывает доступ к огромной доле лабораторных файлов.

## 10. Ноль React-тестов — экзистенциальный риск

### Что сейчас

112 Python-тестов, 0 React-тестов. 28 компонентов отвечающих за визуализацию ДНК, UI праймеров, конфигурацию сборки — без автоматической проверки. Один рефакторинг может молча сломать карту конструкта или таблицу праймеров.

### Vitest, не Jest

Для Vite-проекта Vitest использует общий `vite.config.ts` (не нужна дублирующая конфигурация сборки), стартует **2–10× быстрее** (одна команда сообщила 18.7с → 1.8с), нативный ESM без Babel/ts-jest. `happy-dom` как окружение по умолчанию (3–10× быстрее jsdom).

### Playwright, не Cypress, для E2E

Playwright обогнал Cypress по npm-загрузкам с июня 2024. Бесплатная параллелизация, нативная кросс-браузерность, out-of-process архитектура (35–45% быстрее в параллельном CI), поддержка multi-tab.

### Property-based тесты — уникально сильны для ДНК-операций

`fast-check` (JavaScript) и `hypothesis` + `hypothesis-bio` (Python) для проверки инвариантов на любых последовательностях:

- `reverseComplement(reverseComplement(seq)) === seq` (инволюция)
- `calculateTm(seq) === calculateTm(reverseComplement(seq))` (симметрия Tm)
- `gcContent(seq)` всегда в [0, 1]
- Трансляция CDS длины 3n даёт белок длины n

### 8-недельный roadmap от 0 до полного покрытия

| Фаза | Недели | Тестов | Фокус |
|---|---|---|---|
| Фундамент | 1–2 | ~30 | Vitest + RTL + happy-dom. Unit-тесты всех чистых функций: reverseComplement, gcContent, calculateTm, validateSequence, координатные трансформы. Property-тесты через fast-check. |
| Компоненты | 3–4 | ~40 | Тесты 10 критичных из 28 компонентов через RTL. Snapshot-тесты SVG-рендеринга конструкта. |
| Интеграция | 5–6 | ~20 | Мок бэкенда через MSW. Тесты многокомпонентных workflow: выбор фрагмента → автодизайн праймеров, модификация конструкта → перерисовка карты. |
| E2E + CI | 7–8 | ~8 | Playwright. 5–8 критичных E2E-сценариев. GitHub Actions: Python (3.10/3.11/3.12) + Node (18/20/22) + E2E. |

**Цель: ~230 тестов** (112 Python + 30 JS unit + 40 component + 20 integration + 8 E2E + 20 Python property-тестов).

**Приоритет: критический.**

## Заключение

У PlasmidVCS три реальных блокера (монолитный state, неточный Tm, ноль фронтенд-тестов) и семь возможностей для оптимизации.

Самая неожиданная находка — насколько дёшево стоят самые влиятельные фиксы: React Compiler ставится меньше чем за два часа, `primer3-py` заменяет кастомную Tm-функцию за полдня, 30 Vitest-тестов на утилитарные функции пишутся за два дня.

Миграция на Zustand — самая большая разовая работа (2–3 дня), но она разблокирует все нижестоящие оптимизации производительности.

Golden Gate оказался более зрелым чем ожидалось — MIT-лицензированная тройка `tatapov` + `GoldenHinges` + `DNA Cauldron` от Edinburgh Genome Foundry даёт качество дизайна уровня публикации «из коробки».

Для CRISPR ключевой инсайт: линейная модель CRISPRscan даёт легитимный скоринг в 50 строках Python без зависимостей — ML-фреймворк не нужен для инструмента клонирования.

Выполнить критические пункты первыми — остальное последует естественно.
